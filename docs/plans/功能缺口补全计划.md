# 功能缺口补全计划

> 审计日期：2026-07-14  
> 审计范围：docs/plans 下全部 16 份规划/分析文档，对照 backend/app、backend/alembic、backend/tests、frontend/src。  
> 边界：本文只登记缺口与补全顺序，不修改业务代码、OpenSpec 或当前大纲重构文件。

## 1. 结论摘要

当前项目不是“多数功能都没做”，而是呈现三种状态：基础生成链路已能运行；若干高级模块只有文件或局部类、没有接入真实流程；前端、持久化、恢复能力和测试覆盖明显落后于规划文档。

最需要先锁定的语义是：大纲必须按章节滚动，不按卷滚动。当前核心实现已经以最多 5 章为一批：backend/app/services/outline_rolling.py:5、69-104；backend/app/services/outline_planner.py:190-256；前端按钮也显示“规划接下来 5 章”：frontend/src/pages/ProjectPage.tsx:437-447。卷只应是稳定的结构契约和章节范围边界，不应成为一次生成整卷细纲的执行单位。

“卷会不会不完整”的答案：允许暂时未展开完整，但不能结构不完整。初始化时应先保存全书卷契约、章号范围和卷末锚点，随后逐批补章节；当前代码已在 backend/app/services/outline_planner.py:81-140 保存全部卷契约，并在 335-337 将卷标记为 planning/detailed。需要补的是跨卷、续跑、并发、失败恢复和前端状态的验收，确保未完成卷始终可继续且不会丢章、重章或越界。

## 2. 状态定义

| 状态 | 判定 |
|---|---|
| 已实现 | 主路径已接入，数据可持久化，有基本接口或页面 |
| 半实现 | 存在类/文件/接口，但未接入主流程、未持久化或只有内存实现 |
| 未实现 | 规划中的核心文件、表、接口、页面均不存在 |
| 行为偏离 | 能力存在，但执行单位、接口语义、恢复策略或页面行为与规划/当前需求不一致 |
| 缺测试 | 实现存在，但无足以证明验收条件的自动化测试 |

## 7. 建议实施顺序

1. 先锁定章节滚动契约和完整性测试：保留卷契约，但所有继续生成均按章批次。
2. 接通持久化检查点、可靠重试、熔断、指标和预算，保证长任务可恢复且成本可控。
3. 实现 DHO 候选版本与 Diff，和章节滚动、章节版本共同形成“人工修改后安全续写”闭环。
4. 补角色卡、Memory 持久化、Bible 版本联动和质量 HITL，解决长篇一致性。
5. 拆分并补齐前端页面/状态管理；章节滚动作为主入口，卷仅展示结构、完成度和追加边界。
6. 最后补全自动化测试、CI、离线冲突协议和旧文档状态同步。

## 8. 每个缺口关闭时的统一 Definition of Done

- 规划中的数据结构有迁移且可回滚。
- API 有权限、错误码、幂等和并发行为定义。
- 后端主流程真实调用，不以“文件存在”视为完成。
- 前端有可到达入口，并处理 loading/error/retry/refresh。
- 至少有定向自动化测试覆盖成功、失败和恢复路径。
- 原规划文档同步实际状态，避免规划态再次被误认为实现态。
