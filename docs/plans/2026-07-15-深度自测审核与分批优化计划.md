# 深度自测审核与分批优化计划

日期：2026-07-15

实施状态：**批次 0-6 已完成并通过最终自动化验收（2026-07-15）**。

说明：第 1-7 节保留实施前的审计基线、风险与设计决策，便于追溯“为什么修改”；第 8 节记录实施后的当前状态与最终证据。判断项目现状时以第 8 节为准。

## 1. 结论

当前系统不是不可用原型：项目、大纲滚动、场景写作、规则审校、重写和保存形成了基本链路。
但它还不是可安全运行的长篇小说多 Agent Pipeline。多个高级模块停留在“服务文件或迁移存在”的状态，未进入真实写作主流程，也没有足够的自动化测试证明行为。

当前可确认的主流程：

```text
SceneConstraint -> WriterAgent -> ReviewerAgent -> 最多 3 次重写 -> Scene 保存
```

不应按“已交付”计算的能力：DHO 重规划、可恢复长任务、角色卡、持久化记忆、项目级风格档案、Bible 自动演进、质量报告/HITL、成本和预算控制、章节正文版本分支。

## 3. 审核发现

### P0：阻断安全或生产使用

1. 写作相关接口缺少项目归属校验。
   - `app/api/writing.py` 的 write、write-auto、write-chapter、save、review 按 `scene_id` 查询，但未验证 `current_user_id` 是否拥有项目。
   - 风险：登录用户若取得 UUID，可修改他人场景或触发他人成本。

2. LLM 可靠性与成本控制未接入。
   - `CircuitBreaker`、重试、MetricsCollector、BudgetGuard 已存在，但 `app/llm/client.py` 的同步与流式调用没有统一经过它们。
   - 风险：限流/超时不重试，熔断无效，成本和 token 为空，预算无法阻断。

3. DHO 动态重规划未实现。
   - `BranchManager` 仅把章节置为 `needs_replan`；无重规划引擎、候选版本、差异、确认、回滚与 API。

4. 任务恢复不成立。
   - `CheckpointStore` 使用进程内字典；`TaskManager` 重启后将运行任务标为 `orphaned`。
   - 风险：中断后不能恢复，且没有幂等键避免重复调用或计费。

### P1：核心长篇能力未闭环

| 能力 | 当前状态 | 缺口 |
|---|---|---|
| 章节滚动大纲 | 部分实现 | 有批次纯单测，无 90 章、并发、失败续跑集成测试 |
| Context Budget | 未接入 | `ContextBuilder.build_context_with_budget` 未由写作调用 |
| Memory/RAG | 半实现 | 角色记忆返回空、plot/vector 为内存实现、无持久化生命周期 |
| Bible 版本 | 未接入 | 服务和迁移存在，但写作/审校后不会自动演进 |
| 质量评估/HITL | 未接入 | Evaluator 存在，未落库、无队列、无处理入口 |
| 角色卡 | 大部未实现 | 仅 Schema/Prompt，无 Agent、CRUD、持久化、写作联动 |
| 正文章节版本 | 未实现 | `chapter_versions.py` 实际为大纲快照，不是正文版本树 |

补充行为风险：Reviewer 的 LLM 调用失败时回退为空问题；QualityEvaluator 失败时每维默认为 3 分。这会把外部故障伪装为“可通过”的审校或质量结果。

### P2：工程质量与测试证明不足

- `tests/prompt_regression_test.py` 的 pytest 用例只有 `assert True`，未实际驱动 Writer、未持久化 baseline。
- 未覆盖 DHO、Bible 生命周期、质量/HITL、预算、指标、记忆持久化、恢复、写作 SSE 和权限边界。
- 前端可以构建，但没有发现前端自动化测试。
- 多份旧架构文档将文件存在误标为“功能完成”；后续必须以主流程接入和验收测试为完成条件。

## 4. 分批优化次序

原则：先验证上下文是否真正进入写作请求，再修安全和可靠性；随后建立可版本化、可冻结的数据基础，再建设恢复与重规划能力。每批完成后才进入下一批，避免继续堆积未接入模块。

### 批次 0：建立真实基线

范围：修复测试环境、收敛 lint、建立状态矩阵，并验证 ContextBuilder 是否真正接入 WriterAgent。

- 以 Poetry/项目虚拟环境安装并锁定测试依赖。
- 修复 `retry.py` 未定义变量及阻断性 lint 错误。
- 将现有计划文档中的“规划/已实现/未接入/废弃”状态统一。
- 建立最小 CI：后端编译、Ruff、pytest；前端 typecheck/build。
- 用可脱敏的测试项目抓取一次 Writer 的实际 LLM 请求，确认其中包含约束卡、Bible、前情摘要、活跃伏笔和记忆上下文；禁止在生产日志记录正文或用户敏感内容。

验收：完整 pytest 可收集；Ruff 无 F821/F811；CI 在干净环境可运行；ContextBuilder 具有真实主流程调用点和集成测试，而非只有定义。

### 批次 1：安全与 LLM 统一入口

范围：所有写作相关路由授权、所有 LLM 调用的可靠性包装。

- 为 scene/chapter/project 查询统一加入 owner 校验。
- 在 LLMClient 统一接入错误分类、重试、熔断、调用指标、调用前后预算检查。
- 流式异常、取消和超时均记录失败与已用 token。

验收：跨用户访问返回 404/403；超时、限流、熔断、预算超限和流中断均有定向测试；管理端能查询实际调用数据。

### 批次 2：Bible/Memory 持久化与上下文注入

范围：让长篇事实和记忆成为可版本化的数据源，并真正注入写作。

- 持久化 Bible 版本、角色记忆、情节状态和向量索引生命周期。
- 将 ContextBuilder 接入写作入口，按场景构建并注入 Bible、前情、活跃伏笔、检索记忆和预算分配结果。
- 固定 ContextBuilder 的输出 Schema、版本标识和降级语义，作为后续 ContextSnapshot 的输入契约。
- 写入/删除时维护数据库与向量索引的一致性，支持重建与降级。

验收：第 50 章可检索并使用第 3 章关键事实；Writer 的实际请求包含所需上下文；数据源或向量库不可用时任务明确降级或失败，不伪装为完整上下文。

### 批次 3：任务持久化、ContextSnapshot 与章节滚动可靠性

范围：长任务可恢复、章节批次幂等、写作状态可追踪。

- 将检查点迁移到 Redis 或数据库，记录输入版本、阶段、内容偏移、重试次数与幂等键。
- 增加 retry/cancel/resume API 和安全的重启语义。
- 收口“规划接下来 N 章”作为唯一续写契约，旧卷展开接口降级或移除。
- 覆盖 90 章、跨卷、重复请求、并发请求、失败后续跑。

验收：服务重启后任务可恢复或明确 fail-safe；章节始终为无重复无空洞的连续前缀；重复请求不重复写入或计费。

### 批次 4：正文版本与 DHO 重规划闭环

范围：人工修改后可安全续写。

- 建立不可变 chapter content version、active version 指针和分支 API。
- 实现 DHO 候选大纲、影响范围、章级 diff、确认、拒绝、回滚和乐观锁。
- 只允许修改指定章之后的未锁定章节。

验收：人工编辑不会覆盖 AI 原稿；重规划不触碰已写章节；并发确认产生可处理的版本冲突。

### 批次 5：质量/HITL、角色卡与项目风格

范围：在稳定的上下文、任务和版本底座上扩展可审计的质量能力。

- 章节确认后触发质量评估落库和 HITL 队列。
- 补角色 CRUD、角色对话模拟、口吻/动作注入与漂移审校。
- 增加项目级风格档案、版本及 Writer/Reviewer 同版本引用。

验收：第 50 章可检索并使用第 3 章关键事实；低质量章节进入可处理 HITL；角色和风格变化可审计并影响生成。

### 批次 6：产品入口、回归与发布质量

范围：前端工作台、端到端验收、Prompt 回归。

- 补齐角色、审校、任务、版本、DHO Diff 的前端入口和统一 SSE/task store。
- 将 Prompt 回归改为离线结构测试加独立标记的真实模型验收，baseline 持久化。
- 建立后端 unit/integration/e2e 分层及前端组件/Hook 测试。
- 补离线编辑冲突协议：版本号、409、合并/另存版本、重试队列。

验收：完整“建项目 -> 滚动规划 -> 写章 -> 审校 -> 质量/HITL -> 人工修改 -> DHO -> 继续写作”场景可重复执行。

## 5. ContextSnapshot 与恢复语义

### 5.1 冻结输入而非读取最新状态

任务创建时必须生成不可变的 `ContextSnapshot`。恢复任务只使用该快照；新的生成任务才读取最新 Bible、记忆和伏笔状态。

建议 Schema：

```text
constraint_card_snapshot
bible_version_ids                 # entry_id -> bible_entry_versions.id
memory_retrieval_frozen           # 检索结果全文及来源
foreshadowing_snapshot            # active/resolved 状态
chapter_outline_frozen            # 事件链和大纲版本
context_budget_allocation         # 已分配/丢弃分类
prompt_version                    # 可解析的模板版本
model_id + sampling_parameters    # 模型、温度等调用配置
```

`prompt_version` 不能只保存 git hash：恢复时必须能取得对应模板。实现可选版本化 Prompt 注册表，或在快照中保存已渲染 Prompt/模板正文及其 hash。

冻结输入保证依据一致和可审计；它不保证非确定性 LLM 产出逐字可重复。必须记录模型版本、采样参数、实际输出与调用成本。

### 5.2 中断恢复策略：整场景重写

流式输出中断后，禁止按 token offset 拼接新旧文本。模型 API 通常不能精确续流，拼接会产生语气、事实和时间线断裂。

恢复流程：

```text
加载 ContextSnapshot
-> 保存中断草稿为不可变尝试记录
-> 清空前端 stream buffer，状态为 recovering
-> 使用冻结输入重新生成完整场景
-> 完整审校后保存新草稿/确认版本
```

前端必须显示“正在重新生成整个场景”，而不是“继续生成”；`recovering` 时替换 buffer，不追加旧 token。场景建议控制在 800-1200 字，降低整场重写成本。

### 5.3 成本和预算语义

所有调用都必须记录实际 token 与成本，并计入项目总预算。恢复调用不能完全排除在场景预算之外，否则反复中断可绕过成本上限。

```text
call_type: initial | retry | recovery
scene_initial_budget: 首次生成额度
scene_recovery_allowance: 恢复重写额度
max_recovery_attempts: 最大自动恢复次数
```

当恢复额度或最大次数耗尽时，任务进入 `needs_human_review`，不再自动重写。`call_type` 用于审计、告警和预算分类，不用于隐藏真实成本。

### 5.4 中断草稿保存

在正式章节版本体系完成前，检查点必须保存 `partial_draft_text` 或关联不可变 draft-attempt 记录，以便审计和人工比较。建议优先使用独立的 `scene_draft_attempts` 表，而不是让检查点表长期承载无界正文；该表至少包含 `scene_id`、`task_id`、`attempt_type`、`content`、`context_snapshot_id`、`created_at`。

## 6. 每批统一完成定义

- 变更有迁移、回滚和权限边界。
- 主流程有真实调用点；文件或类存在不算完成。
- 成功、失败、重试/恢复路径至少各有一个自动测试。
- 前端有可到达入口、loading/error/retry 状态。
- 文档状态随实现更新，不再把规划态标为完成。

## 7. 暂不建议做的事项

- 不要先扩展新的 Agent 或 Prompt 模板。
- 不要先做复杂前端视觉重构。
- 不要把 Prompt 回归接真实 LLM 作为默认 CI。
- 不要在 DHO 与章节版本未完成前允许自动覆盖后续大纲。

### 8.2 已落实的关键设计决策

- PostgreSQL 是 Bible、Memory、情节线、任务、ContextSnapshot、大纲版本、正文版本、质量报告和 HITL 的事实源；旧的进程内假实现已删除。
- `ContextSnapshot` 冻结约束卡、Bible version IDs、记忆检索结果、伏笔、情节线、大纲、预算分配和 Prompt 版本；恢复不读取“当前最新状态”。
- 流式调用只允许在首 token 前自动重试。首 token 后失败保存中断草稿，并使用冻结快照整场景重写。
- `call_type` 区分 `initial/retry/recovery`；恢复成本计入项目总预算，并受独立 allowance 与最大恢复次数限制。
- Reviewer 和 QualityEvaluator 失败分别返回 `error/unavailable`；质量不可用时自动进入 HITL，禁止静默默认分或空问题假通过。
- DHO 只修改受影响范围内的未来未锁定章节，批准前保存候选和 diff，拒绝不会改变活动大纲。
- Prompt 回归使用持久化离线 baseline；真实模型验收标记为 `external_llm`，默认 CI 不消耗模型 token。
