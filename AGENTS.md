# AI Agent 行为规范

> 本项目 AI Agent 协作指南，定义 Agent 调用规则与职责边界

---

## 一、Agent 调用原则

### 1.1 调用层级

```
┌─────────────────────────────────────────────────────────────┐
│                      用户 (User)                              │
├─────────────────────────────────────────────────────────────┤
│                   Cursor / Claude Code                        │
│         (规划 Agent、审查 Agent、调试 Agent)                   │
├─────────────────────────────────────────────────────────────┤
│                     Agent 层 (子 Agent)                       │
│  世界观Agent │ 大纲Agent │ 细纲Agent │ 写作Agent │ 审校Agent   │
├─────────────────────────────────────────────────────────────┤
│                     工具层 (Tools)                            │
│   文件读写 │ Shell执行 │ Web搜索 │ 代码审查 │ 知识库检索        │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 Agent 分类

| Agent 类型 | 职责 | 调用场景 |
|-----------|------|----------|
| **规划 Agent** | 分解任务、制定执行计划 | 复杂任务开始前 |
| **审查 Agent** | 代码审查、规范检查 | 代码变更后、提交前 |
| **调试 Agent** | 定位问题、提供修复方案 | 错误排查、异常处理 |
| **写作 Agent** | 小说内容生成 | Phase 2-5 |
| **专业子 Agent** | 特定领域任务 | 按需调用 |

## 二、Agent 调用规范

### 2.1 调用触发条件

**必须调用 Agent 的场景：**
- 复杂的多步骤任务（>3 个子任务）
- 需要深入探索代码库的诊断任务
- 安全审查或 Bug 追踪
- 跨多个模块的架构变更

**直接执行即可的场景：**
- 简单的文件读写（<5 个文件）
- 单文件的代码修改
- 明确的问询（needle query）
- 已规划好的简单任务

### 2.2 调用方式

```markdown
使用 Task 工具调用子 Agent：

Task(
    description: "任务简述（3-5 词）",
    prompt: """
    详细的执行指令，包括：
    1. 任务目标
    2. 上下文信息（必要的文件路径、相关代码）
    3. 输出要求
    4. 约束条件
    """,
    subagent_type: "generalPurpose | explore | reviewer | ...",
    run_in_background: false  // 默认同步，复杂任务才后台
)
```

### 2.3 上下文传递

```markdown
传递给子 Agent 的必要信息：
├── 任务目标（必须）
├── 相关文件路径列表（必须）
├── 关键代码片段（可选，按需）
├── 约束条件/边界（必须）
├── 输出格式要求（必须）
└── 后续处理方式（必须）
```

## 三、Agent 职责边界

### 3.1 Cursor 主 Agent 职责

- 理解用户需求
- 制定执行计划
- 调用合适的子 Agent
- 整合子 Agent 输出
- 最终验证和交付

### 3.2 子 Agent 职责

| Agent | 输入 | 输出 | 副作用限制 |
|-------|------|------|-----------|
| explore | 搜索目标 | 搜索结果 | 只读 |
| reviewer | 代码变更 | 审查报告 | 只读 |
| security-review | 代码变更 | 安全报告 | 只读 |
| bugbot | Bug 描述 | 修复建议 | 只读 |
| generalPurpose | 任意任务 | 任务结果 | 按需 |

### 3.3 禁止行为

```
❌ 禁止在子 Agent 调用中嵌套子 Agent（除非明确需要）
❌ 禁止向子 Agent 传递完整对话历史
❌ 禁止在 Agent 调用中包含 API Key 或密钥
❌ 禁止子 Agent 直接修改非目标文件
❌ 禁止在未查阅文档的情况下使用第三方工具/命令参数
```

### 3.4 第三方命令/工具使用规范

```
使用任何第三方工具、CLI 命令或库 API 前，必须先通过 WebSearch 查阅其官方文档：
- 命令行工具：查阅官方 CLI 文档，确认正确的参数格式和顺序
- 库/SDK：查阅官方 API 文档，确认参数名和用法
- 配置项：查阅官方配置参考，确认合法值和含义

查好后再操作，禁止在参数格式不确定时反复试错。
```

## 四、Agent 协作模式

### 4.1 探索-执行模式

```
用户请求 → Cursor 分析
           ↓
      [探索模式] 如需深入了解
           ↓
      [执行模式] 执行具体任务
           ↓
      [审查模式] 如需质量保证
           ↓
         完成
```

### 4.2 并行任务模式

```
当多个任务相互独立时，可并行调用：

Task 1 ─┐
Task 2 ─┼─→ 并行执行 → 结果整合 → 交付
Task 3 ─┘
```

### 4.3 顺序依赖模式

```
当任务有依赖时，顺序执行：

Task 1 → Task 2 → Task 3 → ...
           ↑         ↑
        依赖1      依赖2
```

## 五、输出规范

### 5.1 Agent 返回格式

```markdown
## [Agent 名称] 执行结果

### 完成状态
✅ 成功 / ⚠️ 部分完成 / ❌ 失败

### 执行摘要
（1-2 句话描述核心结果）

### 详细信息
（详细的执行过程或结果）

### 后续建议
（如有）
```

### 5.2 错误处理

```markdown
## 错误报告

### 错误类型
[具体错误类型]

### 错误位置
[文件:行号 或 组件]

### 错误原因
[简要分析]

### 建议方案
[可行的修复建议]
```

## 六、Context Window 保护

### 6.1 规则

- 原始工具输出必须在沙箱处理
- 优先使用 ctx_batch_execute / ctx_search 等 MCP 工具
- 分析结果仅返回摘要给用户
- 大文件操作分段进行

### 6.2 工具选择优先级

```
0. MEMORY: ctx_search (优先使用知识库)
1. GATHER: ctx_batch_execute (批量执行+自动索引)
2. FOLLOW-UP: ctx_search (后续查询)
3. PROCESS: ctx_execute / ctx_execute_file (处理分析)
```

### 6.3 文件写入

- 始终使用 Write/Edit 工具创建/修改文件
- 禁止使用 ctx_execute 写文件
- 禁止使用 Bash 重定向写文件
